## 2. 커서의 뇌절: "나는 AI를 달래는 유치원 선생님인가?"

가장 답답했던 건 프로젝트의 **근본적인 맥락**을 놓칠 때였습니다.

- **Surface-level Bug Fix의 한계:** 아이디 변경 시 데이터 조회가 안 되는 문제였는데, 백엔드의 `X-Username` 인증 로직을 보는 게 아니라 "변수명을 `name` 대신 `id`로 써보세요" 같은 엉뚱한 소리만 하더군요. 근본 원인은 못 찾고 겉핥기식 코드만 뱉어내는 거죠.
- **무한 루프와 멈춤 현상:** (이미지 1 첨부) 똑같은 말을 반복하며 대화창을 도배할 때는 정말 당혹스러웠습니다.

- **‘파일 단위’로 코드를 짜는 비효율적인 경향** : 특히 프론트에서 심했는데 회원가입 단에 이름 입력 → 생년월일 입력 등 디자인이 일관되어 있는게 있어도 컴포넌트를 분리해서 재사용하지 못하고 페이지를 다 나눠서 파일을 만듦. 뒤로 가기 버튼 위치나 확인 버튼 디자인이 변경되면 두 파일 모두 수정해야하는 정말 안좋은 코드였음. 클로드로 리팩토링하기 (surface-level 개발 : 가장 큰 한계, 특히 유지보수 어려운 코드는 정말 안 좋은 코드니까)

## 5. 클로드로 갈아타게 된 결정적 이유

도전 과제 : 수천 개의 논문을 분석해 영양소 시너지와 부작용 및 효과적인 레시피 산출 AI 엔진 구축 

프로젝트 구조 : 논문 데이터 → Atomic Facts → Rule Mapping → Reduce 데이터 파이프라인 설계

나의 방식 : 읽는 양이 워낙 방대하다 보니까 커서에 소모되는 토큰양이 너무 많았음 월정액이지만 하루에 거의 반을 다 쓸정도로. 
GPT/제미나이로 로직 설계 및 보완점 확인 → 커서로 프롬프트 작성 및 코드 작성 

느낀점 : 1개월만에 mvp를 뽑아냈지만, 가중치를 미세하게 조정하고 레시피 산출 로직을 고도화하려니 커서와의 대화가 너무 파편화되어있었다. 엔진의 전체 설계도를 보고 대화하고 싶은데 커서는 그 많은 부분 중 한 칸을 닦고 있는 느낌 

커서의 한계 : 전체 데이터가 흐르는 규칙이 깨지지 않는 게 중요한데, 부분적인 코드 수정에 집중하기 때문에 전체 파이프라인 일관성을 놓쳐서 고쳤던 부분을 또 고치고 하는 문제점이 있었음. 

클로드의 강점 : 복잡한 룰과 컨텍스트를 기억하고 가중치 조정이나 파인튜닝 시에 전체 로직의 충돌 여부를 확인하는 데 성능이 높음 

- **Before (Cursor + Chat):** GPT에게 로직 물어보기 → 답변 복사 → 커서에 붙여넣기 → 에러 나면 다시 GPT에게 질문 → 무한 반복 (컨텍스트 분절)
- **After (Claude Code):** 터미널에서 엔진 코드 전체를 열어두고 "현재 영양소 충돌 로직에서 가중치 부여 방식이 논문 A의 기준과 안 맞아. 전체적으로 다시 매핑해줘."라고 한마디로 해결.

## 6. Claude Code (CLI)로의 전향: "인턴이 아니라 파트너가 필요하다"

그래서 저는 Claude Code로 갈아타기로 했습니다. IDE의 안락한 UI를 포기하고 검은 터미널(CLI)로 돌아가는 이유는 명확합니다.

1. **자율적인 맥락 파악:** 내가 파일을 일일이 열어주지 않아도, 클로드는 스스로 프로젝트 전체를 탐색합니다.
2. **실행 중심의 에이전트:** 코드를 제안만 하는 게 아니라, 직접 실행해보고 에러를 확인하며 '완성'까지 달립니다.
3. **설계에 집중하는 환경:** 멍청한 도구와 씨름하며 오타를 고치는 시간 대신, 더 큰 아키텍처를 고민할 시간을 벌어줍니다.

## 5. 마치며: 도구에 휘둘리지 않는 똑똑함

"더 똑똑한 도구를 쓰면 내 실력이 줄어들지 않을까?"라는 고민도 했습니다. 하지만 결론은 명확합니다. **멍청한 도구를 붙잡고 시간을 쓰는 게 실력이 아닙니다.** 똑똑한 도구를 활용해 더 큰 문제를 해결하고, AI가 짠 코드를 날카롭게 검수하며 시스템을 설계하는 능력이 현대 개발자의 진짜 실력입니다. 이제 저는 Cursor라는 편한 의자를 떠나, Claude Code라는 강력한 엔진에 올라타 보려 합니다.

### **비지니스 속도에 개발 속도 맞추기 (뒤꽁무니 따라가기)**

스타트업의 특성상 코드의 품질보다는 만들어내는 속도가 중요하다. 비지니스의 속도에 따라가야하는데 혼자 모든 걸 하려고 하니 많이 벅찼던 것 같다. 출시 날은 정해져있는데 모든 일이 그렇듯 단축할 수 없는 base time이 존재하고 개발도 그 타임이 긴 편인데 이 base time으로도 촉박했다. 커서한테 기능을 설명하고 코드를 구현하고 디버깅을 하기에 급급해서 유지보수하기 좋은 코드, 깔끔한 코드는 신경 쓸 겨를이 없었고 결과는 이렇게 나왔다. 

### 좋은 팀과 나라는 사람의 괴리

솔직히 우리팀은 평균 나이가 적고 다 학부생임에도 불구하고 성과가 좋았다. 여러 선배 기업들로부터 인정도 받고 여러 계약을 따내기도 했고 지원사업도 많이 붙었고 어디에 속하든 항상 막내였다. 이렇게 좋은 팀에 있었지만 나의 기술에 대한 확신, 내 지분에 대한 확신이 없었던 이유는 어차피 커서가 짰는데? 내가 직접 코드를 작성하지 않았는데? 디버깅을 할땐 커서한테 물어보는데? 이런 AI에 의존하는 나의 모습 때문에 확신이 없었던 것 같다. (전문지식이 없는 게 한 몫)

### 정체성의 혼란과 과도한 책임감

더군다나 반년전까지만 해도 그저 춤추기 좋아하는, 이것저것 하고 싶은 게 많은 컴공생이었고 아이들 좋아하는 초등영어강사였는데 갑자기 하루아침에 일과의 대부분이 개발이 되어버리니 그 간극을 못 견뎌서 힘들기도 했고 사람들이 사용하는 서비스이다 보니 에러가 나는 것도 나의 책임, 그걸 빠르게 고쳐야하는 것도 나의 책임이라 생각했다. 학교에서 수업듣고 친구들과 점심 먹다가 에러 발생해서 택시 잡고 회사 가면서 디버깅하기도 했고 출시 n일 전까지는 다 되어야 한다는 생각에 10일 연속으로 대표님과 회사에서 밤을 새기도 했다 (공용사무실에는 리클라이너 의자가 딱 두개 있다)

### AI만 쓴 건 아니긴 해~

하지만 우리 팀의 메인 서비스(안드로이드 키오스크 앱)는 커서를 20% 정도만 썼다. 대부분의 파일들은 코드 한줄한줄 내가 다 넣어가면서 만들었고 설정 파일이나 디버깅을 커서로 돌렸다. (아무래도 메인이다보니 코드가 그래도 중복을 최소화한 유지보수 가능한 것으로 만들고 싶었어서 시간이 없는데도 이렇게 했다, 키오스크 앱이라 가능했다 모바일 앱이었으면 시간내에 못했을 것)

### 운도 실력이다 라는 말에 대한 나의 생각

literally 운은 사주상 운세 흐름이 좋아서 나에게 좋은 기운을 주는 거고 실력은 내가 내 의지로 노력해서 쌓아온 전문 지식이나 지혜를 말하는거잖아 둘이 다름. 근데 그 운을 본인의 것으로 가져오는 건 실력이 맞는 것 같다. 아직도 생각한다. 그때 언니가 나한테 그 공고문을 보여줬을때 내 머리를 지배하던 ‘이거 했다가 이 서비스 망하면 어떡해 내가 잘 못하면 어떡해?’라는 생각을 계속 가지고 있었다면 이런 경험들 이런 생각들 다 못했을 것. 운 좋게 공고문을 봤고 일단 내 포트폴리오만 내 보자 판단은 그쪽이 하겠지 라는 용기를 가진 건 내 의도였고 그 포트폴리오를 그동한 준비해왔던 건 내 노력과 실력이었다. 

### 그래서 지금은?

사실 그때 당시에 앱, 웹, 서버 이렇게 만들어서 테스트베드 출시 했을때는 아 내가 해냈다 라기 보다는 에러 언제 뜰라나 라는 불안감이 먼저였다. 하지만 점점 내가 맡는 태스크가 늘어나고 참석하는 미팅도 다양해지고 다른 회사 사람들도 만나고 의견을 내는 과정이 생기다보니 자연스레 자리를 잡았다고 느껴진것같다. 또한 세상이 빠르게 바뀌면서 ‘코드를 직접 타이핑하는 게 실력’이라고 여기던 개발자 시장에서 ‘AI를 활용해 복잡한 문제를 해결하는 게 실력’이라고 바뀌는 걸 보면서 AI를 사용하면서 서비스를 만든 내가 그렇게 잘못된 것은 아니구나 라고 생각이 들었던 것 같다.
