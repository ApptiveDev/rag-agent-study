evidence 에서 rule로 가는거에서 rule이 제대로 안나오길래 중간에 의미 단위로 끊는 atomic facts 레이어를 넣으려고 한건데 알고보니까 assoication rules는 5-tuple중에 2-tuple만 반영하고 있었음 (ingredient, state_tag) 

실제 5-tuple은 final_guidelines(just 가시화 위한 가이드라인)에서만 쓰이고 2-tuple만 쓰이고 있었음. 이게 버그는 아니지만 5-tuple vs 2-tuple 그룹화하는 기준이 다르기 때문에 final_guidelines와 최종 추천 결과가 상충될 수 있다는 점이 문제임

→ 이렇게 되면 atomic_facts 를 넣으면서 분리하려고 했던 것들 (population_segment, dose_route, outcome_normalized, claim_subject)를 전부 날리게 되는 구조임, 결과적으로 evidence→2-tuple grouping은 atomic facts를 도입하기 전의 상태와 크게 다르지 않는 것. 

final_guidelines는 성분별 리포트(테스트위함)를 생성할때만 읽히고 추천 엔진에서는 쓰이지 않음. 결론적으로 이걸 유의미하게 쓰기 위해서는 atomic_facts를 입력으로 받도록 해야하는것 (rule이)

        - 장소마다 필요성 다름
    - 소화 상태 : 소화가 안 좋은 ‘날’에는 고용량 철분이나 비타민C같은 자극 성분을 줄여야함. 프로바이오틱스 추천을 앞당길 수 있기 때문에
    - 오늘 수분 섭취 : 탈수 상태에서는 전해질 우선순위 오리고, 크레아틴 같은 일부 성분은 충분한 섭취 없이 복용하면 신장 부담이 되기 때문에 중요 → 질문 고민 (보수적으로 해도 ok)
    - 전날 음주 여부 : 고용량 비타민 A나 오메가3 고용량 등은 급성 음주 이후에 간 독성 시너지나 항응고 효과를 증폭시키는 등 음주 시 피해야하는 성분이 있기 때문에 constraint 용으로 필요 → constraint (회복하는 걸 더 기대함)
    - 추가하면 되고 + state_tag 배치

### 260515 회의

1. E2R 엔진 검증에 대한 피드백
    1. (제일 중요) 이 시스템은 두 단계가 완전 분리되어 있는 구조임
        1. LLM이 실제 추천 시점에도 관여하는 end-to-end 시스템이 아니라
        2. Step 1 : 오프라인 배치 (1회성) - LLM이 pdf 읽고 evidence 추출하기 + rule 생성하기
        3. Step 2: 실시간 추천 - 사용자 입력을 통해 니즈벡터를 계산하고 rule 조회해서 추천 결과 내기 
        4. T3 일관성 테스트 아예 새로 파야함(설계 자체가 다름), T4 구조적 재설계 필요 
    2. T3. 일관성 테스트 - Jaccard ≥0.99 너무 낮음
        1. 추천 엔진은 LLM을 실시간으로 호출하는게 아니라 이미 추출된 것을 바탕으로 순수 rule 기반이기 때문에
        2. 동일 입력 → 동일 출력이 보장되어야 함. 100%가 되어야 함 
        3. temperature 조정 하는건 제거 하는걸로 (실제 측정 대상이 없음)
        
        ⇒ 동일 논문 텍스트 → LLM 추출 5회 반복 → evidence 필드별 일치율을 확인하는 쪽으로 해야함 (study_type, outcome_normalized, effect_direction, confidence_score) 
        
    3. T4. 5-fold CV가 아님 지금 5000편 논문이 아니라 1,000편 논문 상태다. 근데 이게 문제가 아니라 rule coverage는 달라질수밖에 없음 subset 마다 
        1. 성분 분포가 달라져서 표준편차가 5%p 이내로 나오기 어려움. 논문 수를 조정하거나 표준편차를 조정하거나 
        2. 5-fold 검증을 하려면 rule 생성 코드를 5번 다른 subset으로 돌려야 함. 기술적으로는 가능하지만 fold당 200편인 지금 규모에서는 통계적 의미가 없음.
        3. 또한 검증 fold로 무엇을 측정하는지 정의 필요 - held-out 논문 클레임이 생성된 rule에 포함되는지, 이 설계는 rule 커버리지 테스트이지 추천 정확도 테스트가 아님
        4. **fold마다 coverage가 달라지는 건 시스템의 불안정성이 아닌 당연한 결과임**
            1. 오메가3 논문이 fold 1에 몰려있으면 오메가3 rule 만 많이 나오는거고. 이걸 비교해서 얻는 게 없다
            2. k-fold는 ML에서 “학습 데이터로 본 적 없는 레이블을 예측할 수 있는가”를 텍스트하는 건데 이 시스템엔 예측 대상 레이블이 없음. 측정할 가치도 없고
        
        ⇒ Rule Score 안정성 측정으로 (부트스트랩 방법)
        
        - 논문 중 무작위 20%를 제거하고 rule 생성 코드를 돌려서 핵심 ingredient-outcome 쌍의 rule_score 변동이 얼마나 되는지 N회 반복해서 표준편차
        - 오메가3 + cardiovascular_risk 가 전체 논문에서는 rule score가 0.82였고 80% bootstrap 20회 평균으로는 0.79가 나왔다면 0.04가 표준편차(안정적)
        - 마그네슘 + sleep_problem 이 전체 논문에서는 0.61 이었고 80% 부트스트랩 20회 평균 냈더니 0.58 나왔다면 0.12가 불안정 (논문 수 부족으로)
        - 코엔자임, 크롬, 비타민, 전해질 폴더 논문들 수 적음 (20개이하) - 부트스트랩 취약
