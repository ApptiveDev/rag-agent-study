## 🗂 오늘 작업

고객정보 조회 api 호출 시 offset과 limit 확인하는 습관 장착하기…

## ⚙ 기술 / 개념

🐛 트러블슈팅

- 문제: 고객정보와 주문 데이터 동기화 과정에서 고객정보의 성별/생년월일이 동기화되지 않는 문제 발생
- 원인: 아임웹 API 응답값을 확인해 봤는데 total_count는 아임웹 페이지에 있는 고객 그대로 들어오는 것으로 확인함. 디비 접속해서 회원 정보 업데이트 안된 사람들 확인해봤더니 성별과 생년월일이 아임웹에서 제대로 들어온사람은 딱 100명이었다.. 100명 초과가 됐을 때 아임웹 회원가입 시 설문을 통해 성별을 입력받는 시기가 약간 겹쳤었어서 이 설문 때문에 정보가 제대로 안들어오는건지, 백엔드 코드 문제인지 헷갈렸는데 아임웹 api 응답값 직접 확인해보니 제대로 들어오고 있음에도 불구하고 우리 서버에 적용이 안되는 경우가 있었음.
- 해결: 아임웹 api reference를 열심히 뒤진 결과 offset과 limit 요청 파라미터로 데이터가 페이지네이션 돼서 들어오고 있었음. offset인 걸 모르고 page로 임의로 넣고 있었는데 이게 아니었지… limit 최댓값이 100이었고 아 이것 때문이구나 해서 데이터 다 받을때까지 offset 하나씩 늘려서 계속 데이터 받는 방식으로 동기화 로직을 변경했다.

## 🧠 인사이트

외부 API 연동 시에는 응답 규격과 목록 요청 파라미터, 페이지네이션 정책을 꼭 확인하자. 

아임웹에서 고객 정보 조회/주문 조회 두개 쓰고 있었는데 주문 조회는 제일 최근꺼 몇개만 받아오고 우리 디비에 업데이트하는 식이었어서 offset, limit 이 소용 없었음. 고객 조회는 제일 처음 사람부터 보내줘가지고 이게 중요했다.
