### 멀티 에이전트 구조 채택 이유 

기존에는 paper → evidence llm이 추출하고 생성한 rule → 바로 추천까지 가서

추천 결과에서 임상학적으로 잘못 나온게 있는지 하나하나 확인했음 → 지피티한테 페르소나 먹여서 제대로 나오는지 확인했는데 이렇게 하니까 의약품이 들어가는 경우도 있고 맥파속도 증가(안좋은거)인데 해당 성분이 어떤 state를 변화하는데 positive 하게 작용했다고 잘못 evidence direction 필드가 채워져서 룰 품질이 너무 손상되는 경우가 꽤 있었다. 그래서 임상 지식을 빡세게 걸고 추후 필터링 로직에서 잘못 결과가 나올때마다 각 경우를 사전에 정의해서 추천때 막는걸로 계속 했는데

이건 진정한 자동화가 아니고 코드도 더러워지고 유지보수할때 너무 힘들것 같다는 생각을 했음 

지금 문제인건 evidence 필드 누락 혹은 할루시네이션 발생 
→ 이로 인한 데이터 품질 하락 및 rule 생성 시에도 결합하는 구조에서 supports_to_target만 반ㅇ여하고 interacts_with_target 은 아예 rule로 안들어가는 evidence로 처리하는 등 할루시네이션으로 인한 룰 품질 저하를 막기 위해 일부로 설정한 몇몇 보수적인 로직때문에 rule 이 적게 나오는 경우 발생
→ 데이터 품질 저하 뿐만 아니라 rule 생성에서도 데이터가 많이 탈락되는 게 보였고 이런 사전 정의나 필터링을 아무리 걸어도 rule 이 제대로 나오는게 맞는지에 대한 확신은 여전히 부족했음

⇒ 멀티 에이전트 구조로 가서 이중으로 검증하자 (추출보다 중요한게 raw data를 정형화했을때 잘 들어갔냐에 대한 검증이라고 생각했고)

evidence 필드에 대한 데이터 적합성 검증 + rule 자체의 임상학적 커버력 검증 

두개를 추가했고 이게 성능 및 추천 결과상 어떻게 작용했냐면

- 새 논문 추출시 사전이나 필터링 로직 추가한 횟수가 몇번에서 → 몇번으로 절감
- 이로 인한 추천 결과 품질이 어느정도 향상 (정확도)

Adversarial Injection 을 하기로 결정한 이유

모든 것을 에이전트가 검증하고 그 검증한 걸 모두 믿기에는, 할루시네이션 때문에 검증 에이전트를 추가한 거기 때문에 영원히 할루시네이션의 두려움에서 벗어날 수 없음 →인간의 수동 검증을 필수적으로 들어가게 하되, 수동 검증시 발견한 오류 패턴을 프롬프트에 집어넣어서 계속해서 검증하는 패턴을 추가해나가는것 (FN Rate가 안정될때까지)

수동검증을 하는 목적은 2개가 있는데

1. 캘리브레이션 : validator 가 못 잡는 새 오류 패턴이 있는지? - 새 패턴 발견이 멈출 때 수동 검증 중지
2. 모니터링 : 현재 FN rate 가 허용 수준인지? - 수치가 안정될 때 수동 검증 중지

FN Rate 수치는 어떻게 하면 될까 (FN rate 95% 신뢰구간 측면에서)

1. 20개 검토, 0개 오류 → FN Rate가 14% 정도 → 너무 불확실
2. 60개 검토, 0개 오류 → FN Rate가 6% 정도 → 아직 부족
3. 100개 검토, 0개 오류 → FN Rate가 4% 정도 → FN rate 5% 이하를 목표로 하면 80개 연속 clean이 나와야 통계적 근거가 생김

### 확인 필요 필드

- study_type : evidence 의 점수를 결정짓는 필드이기 때문에
- state_tags : state tag가 “억지없이” 수용가능할 정도로 들어갔는가
- ~~human_review_status~~
    - ~~unreviewed : validator(검증 llm)가 approve(문제없음)으로 남긴 것 → validator가 놓친 오류가 있는지~~
    - ~~validator_correct : 오류 있어서 (direction, 의약품, 인과관계 혹은 state_tag 등의 누락) 고친 것 → validator가 수정을 제대로 했는지? 확인~~
- ingredient_applicability : ingredient_name (해당 근거에서 중심을 두고 있는 성분)과 state_tag와 직접적 관련이 없으면 unrelated로 되어있는가 (관련이 없는데 direct로 되어 있진 않은가)
- claim_subject : 성분-효과의 주체
    - substance_intervention : 해당 성분이 상태를 변화시켰을때 (룰이 만들어지는 evidence)
    - covariate_pattern : 공변량 패턴으로 성분이 성분을 변화시키는 등 다른 변수가 있는 경우 (룰 생성 x)
    - behavior_exposure : 특정 행동을 통해 상태가 변화한 경우 (룰 x → just 매커니즘 노트로 들어감)
- direction : 성분-상태의 관계 벡터
    - positive : 성분이 상태의 상황을 “좋은 방향으로” 변화시켰는가
    - negative : 성분이 상태의 상황을 “나쁜 방향으로” 변화시켰는가
    - unclear/mixed : postive와 negative가 충돌했거나 하나의 벡터로 확정짓기 어려움

paper → evidence → rule
